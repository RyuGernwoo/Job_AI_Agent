## 2. 프로젝트 개요

### 2.1 서비스명

**LectureOps Agent MVP**  
직업훈련 강사가 커리큘럼과 교재를 넣으면, 차시별 강의 초안·실습 과제·평가 문항을 생성하고 강사가 검수한 뒤 문서로 내려받을 수 있는 보조 AI Agent.

### 2.2 해결하고 싶은 문제

원문 문서의 문제 정의를 MVP 기준으로 재해석하면 다음과 같다.

- 직업훈련 강사는 과정별 교안, 실습 과제, 평가 문항을 반복적으로 작성해야 한다.
- NCS 능력단위와 커리큘럼을 근거로 산출물을 만들어야 하지만, 사람이 매번 수작업으로 정합성을 맞추기 어렵다.
- 강사마다 형식과 난이도가 달라 산출물 품질 편차가 생긴다.
- 생성형 AI를 그냥 쓰면 근거 출처가 불명확하고, 저작권·할루시네이션 문제가 생긴다.

### 2.3 MVP 목표

**한 달 안에 시연 가능한 MVP**를 목표로 한다.

MVP의 성공 기준은 “완전 자동 강의 제작 플랫폼”이 아니라, 다음 흐름을 끝까지 보여주는 것이다.

> 커리큘럼 + NCS 능력단위 + 샘플 교재 업로드  
> → 근거 문서 기반 검색  
> → 1개 차시 강의 패키지 생성  
> → 강사 검수·수정  
> → DOCX 또는 PPTX 다운로드

## 3. 명시적 가정과 불확실성

| 구분 | 내용 |
|---|---|
| 작업 기간 | 4주, 개인 프로젝트 |
| 개발자 수준 | AI 전공 졸업생 수준, 상용 서비스 수준의 LMS 구축 경험은 없음 |
| 입력 자료 | 샘플 커리큘럼, NCS 능력단위 텍스트, PDF/DOCX 교재 일부가 제공된다고 가정 |
| LLM | CLOVA Studio·HyperCLOVA X, OpenAI API, 또는 로컬 LLM 중 하나를 선택 가능하다고 가정 |
| 배포 | 로컬 실행 또는 간단한 클라우드 데모 수준 |
| 제외 | 기관 단위 계정 관리, 정식 LMS 연동, 결제, 학습자용 서비스, 자동 채점 운영 시스템 |

불확실한 항목은 다음 단계에서 확인해야 한다.

- 실제 사용할 LLM API가 무엇인지
- 샘플 교재의 저작권과 외부 API 입력 가능 여부
- PPTX 출력이 필수인지, DOCX 출력만으로 제출 가능한지
- NCS 능력단위 데이터를 수동 입력할지, 파일 업로드할지

## 4. 딥 리서치 요약

### 4.1 상용 서비스에서 확인한 기능

| 서비스 | 실제 기능 요약 | MVP 반영 여부 |
|---|---|---|
| [Easygenerator EasyAI](https://www.easygenerator.com/en/features/easyai/) | 문서 업로드 기반 과정 초안 생성, 학습 목표와 대상자 맥락 설정, 질문·플립카드·이미지 등 학습 요소 생성, 지식 베이스 활용 | `문서 업로드 → 강의 초안 생성` 흐름 반영 |
| [Coursebox AI](https://www.coursebox.ai/) | AI course creator, quiz/flashcard/scenario 생성, AI tutor, AI grading, LMS·SCORM·LTI 연동을 제공 | 퀴즈·실습 시나리오 생성은 반영, LMS/SCORM/LTI는 제외 |
| [MagicSchool AI](https://www.magicschool.ai/) | 교사용 AI 도구 묶음, 수업 계획·프레젠테이션·루브릭·객관식 퀴즈·피드백 생성 | 강사용 생성 도구 묶음이라는 제품 방향성 참고 |
| [Articulate Rise](https://www.articulate.com/360/rise/) | 웹 기반 과정 제작, 인터랙티브 블록, AI Assistant, SCORM/xAPI/LMS 게시 기능 | 결과물 UX 참고, LMS 게시 기능은 MVP 제외 |

상용 서비스의 공통점은 다음과 같다.

- 강사를 완전히 대체하지 않고, 초안을 빠르게 만든 뒤 사람이 고친다.
- 문서나 기존 자료를 입력으로 삼아 생성 품질을 높인다.
- 퀴즈, 실습, 프레젠테이션, 과정 구조를 묶어서 제공한다.
- LMS 연동과 SCORM/xAPI는 상용 단계에서 중요하지만, 한 달 MVP에는 과하다.

### 4.2 오픈소스·프레임워크에서 확인한 기능

| 도구 | 실제 기능 요약 | MVP 반영 여부 |
|---|---|---|
| [Dify Knowledge Base](https://docs.dify.ai/en/guides/knowledge-base) | 외부 지식 문서를 업로드하고, 검색 설정·메타데이터·retrieval test를 통해 RAG 기반 앱을 구성 | 빠른 MVP의 RAG 구현 참고 |
| [RAGFlow](https://github.com/infiniflow/ragflow) | 문서 이해 기반 오픈소스 RAG 엔진, 근거 인용, 다양한 파일 형식, API, agent capability 제공 | 기능 벤치마크로 참고하되 직접 도입은 보류 |
| [LangGraph Human-in-the-loop](https://docs.langchain.com/oss/python/langgraph/human-in-the-loop) | agent 실행 중 중단점, 사람 검토, 상태 유지 기반 워크플로우 제공 | 강사 검수 단계의 구조 참고 |
| [Moodle Questions](https://docs.moodle.org/405/en/Questions) | 객관식, 단답형, 서술형, 매칭 등 다양한 문제 유형과 문제은행 구조 제공 | 평가 문항 유형 설계 참고 |
| [H5P Content Types](https://h5p.org/content-types-and-applications) | Course Presentation, Interactive Video, Branching Scenario, Quiz 등 인터랙티브 학습 콘텐츠 유형 제공 | 실습 시나리오·퀴즈 유형 참고 |
| [FastAPI](https://fastapi.tiangolo.com/) | Python 기반 API 서버, 자동 API 문서, 파일 업로드 API 구현에 적합 | MVP 백엔드로 채택 |

오픈소스 쪽에서 얻은 결론은 다음과 같다.

- RAG는 직접 처음부터 크게 만들기보다, Dify·RAGFlow 같은 구조를 참고해 작게 시작하는 편이 낫다.
- Moodle/H5P는 교육 콘텐츠의 표준적인 문제 유형과 실습 콘텐츠 유형을 잡는 데 좋다.
- Human-in-the-loop는 “생성 후 바로 확정”이 아니라 “생성 → 검수 → 수정 → 확정” 상태 모델로 설계해야 한다.
- FastAPI는 API와 문서화가 빠르므로 개인 MVP에 적합하다.

### 6.3 Porter Five Forces

| 요인 | 강도 | 근거 |
|---|---|---|
| 기존 경쟁자 | 높음 | Easygenerator, Coursebox, MagicSchool, Articulate 등 AI 기반 교육 콘텐츠 제작 서비스가 이미 존재 |
| 신규 진입 위협 | 높음 | LLM API와 RAG 프레임워크로 MVP 진입이 쉬움 |
| 대체재 위협 | 높음 | ChatGPT, CLOVA Studio, Google Docs, PPT 템플릿만으로도 강사가 수작업 보조 가능 |
| 구매자 교섭력 | 중간~높음 | 교육기관은 가격 민감도가 있고, 기존 LMS/문서 도구와 비교함 |
| 공급자 교섭력 | 중간 | LLM API, embedding API, 클라우드 인프라에 의존할 수 있음 |

### 6.4 SWOT 분석

| 구분 | 내용 |
|---|---|
| Strengths | NCS·커리큘럼·교재를 함께 입력받아 직업훈련 문서 형식에 특화 가능, HITL 검수 구조 명확 |
| Weaknesses | 개인 프로젝트라 데이터 규모와 UI 완성도가 제한됨, 대형 LMS 연동 경험 부족 |
| Opportunities | AI 교육 과정 증가, 강사 업무 자동화 수요, RAG 기반 출처 표시 요구 증가 |
| Threats | 상용 서비스의 빠른 기능 확장, 교재 저작권 이슈, 생성 결과 품질 편차 |

### 6.6 KSF: 핵심 성공 요인 5개

|---|---|

| 저작권·출처 관리 | 교재 기반 생성물은 출처와 사용 범위 관리가 필수다. |

## 9. 시장·경쟁사 분석

### 9.1 TAM/SAM/SOM 산정 방식

현재 문서에서는 검증되지 않은 시장 규모 숫자를 임의로 넣지 않는다. 대신 KOSENA 양식에 맞춰 산정 산식을 정의한다.

| 구분 | 정의 | 산식 |
|---|---|---|
| TAM | 국내 직업훈련·성인교육 콘텐츠 제작 보조 전체 잠재시장 | `직업훈련/성인교육 강사 수 × 연간 콘텐츠 제작 지출 또는 도구 지불 의향` |
| SAM | AI·디지털·프로그래밍 과정처럼 문서와 실습 생성 수요가 높은 도달 가능 시장 | `AI/디지털 직업훈련 과정 수 × 과정당 강사/운영자 수 × 예상 지불액` |
| SOM | 1~3년 안에 개인·소규모 기관 대상으로 확보 가능한 초기 시장 | `초기 접촉 가능 기관 수 × 전환율 × 월/연 구독 단가` |

데이터 수집 후보:

- 고용노동부·HRD-Net 공개 통계
- 직업훈련기관 과정 수
- AI·디지털 과정 개설 수
- 강사 대상 설문 기반 지불 의향
- 유사 SaaS 가격대

| 축 | 낮음 | 높음 |

## 11. AI 활용·검증 로그 계획

KOSENA 자료는 AI 응답을 그대로 사용하지 않고, 검증·수정·재구성 후 반영하라고 요구한다. 이 프로젝트는 서비스 자체도 AI를 쓰므로, 기획서와 MVP 모두에서 AI 사용 로그를 남긴다.

| 로그 항목 | 기록 내용 |
|---|---|
| 역할 | 강의 설계 보조 Agent, 평가 문항 생성 Agent 등 |
| 입력 데이터 | 과정명, 차시명, NCS 능력단위, 검색된 교재 chunk |
| 요구사항 | 교안/실습/평가 중 생성 대상, 난이도, 문항 수 |
| 출력 형식 | JSON schema, DOCX 템플릿 섹션 |
| 검증 조건 | citation 존재 여부, 정답·해설 존재 여부, 강사 승인 여부 |
| 채택 여부 | 원문 채택, 수정 후 채택, 폐기 |
| 수정 사유 | 근거 부족, 난이도 부적합, 문항 오류, 표현 수정 |

발표자료는 다음 5분 흐름으로 구성한다.

| 슬라이드 | 내용 |
|---|---|
| 1 | 문제 정의: 직업훈련 강사의 반복 산출물 작성 부담 |
| 2 | 고객·시장: 강사/운영자 페르소나와 Pain Point |
| 3 | 경쟁 분석: 상용 서비스와 오픈소스 대비 차별점 |
| 4 | 서비스 컨셉: RAG + HITL + 문서 export |
| 5 | MVP 범위: Must/Should/Won't Have |
| 6 | 데모 흐름: 업로드 → 생성 → 검수 → 다운로드 |
| 7 | 1개월 로드맵과 한계 |

### 12.2 대표 시나리오

1. 강사가 과정명, 차시명, 학습목표, NCS 능력단위를 입력한다.
2. 샘플 교재 PDF/DOCX 또는 텍스트 자료를 업로드한다.
3. 시스템이 자료를 분할하고 검색 가능한 지식 베이스를 만든다.
4. 강사가 “1차시 강의 패키지 생성”을 실행한다.
5. AI가 다음 항목을 생성한다.
   - 교안 초안
   - 실습 시나리오
   - 객관식 평가 5문항
   - 실습형 평가 1개
   - 각 항목별 근거 출처
6. 강사는 생성물을 검수하고 수정한다.
7. 최종 결과물을 DOCX로 내려받는다.
8. 가능하면 PPTX 요약본도 내려받는다.

## 14. 추천 구현 전략

### 14.1 후보 접근법

| 접근 | 장점 | 단점 | 판단 |
|---|---|---|---|
| A. Dify 기반 빠른 조립 | RAG와 워크플로우를 빠르게 구성 | 코드 포트폴리오로 보여줄 부분이 줄어듦 | 발표용 데모에는 좋음 |
| B. FastAPI + Chroma/LangChain 또는 LlamaIndex | 직접 구현 경험을 보여주기 좋음, 구조 설명 가능 | RAG 품질 튜닝 부담 | **추천** |
| C. RAGFlow 직접 도입 | 기능이 강력하고 근거 인용 구조가 좋음 | Docker·리소스·학습 비용이 큼 | 한 달 MVP에는 보류 |

### 14.2 최종 추천

**FastAPI + 간단한 RAG + 강사 검수 + DOCX 출력**을 기본으로 한다.

추천 이유:

- 개인 프로젝트에서 코드와 설계를 모두 설명하기 좋다.
- 원문 요구의 FastAPI, RAG, HITL, 다운로드 기능을 모두 최소 형태로 충족한다.
- Dify/RAGFlow 같은 실제 오픈소스 흐름을 참고하되, 과한 플랫폼 구축은 피할 수 있다.
- 상용 서비스들이 공통으로 제공하는 “문서 기반 강의 초안 + 평가 생성 + 사람 검수” 흐름을 재현할 수 있다.

## 15. 시스템 아키텍처

```text
[강사 입력 화면]
  ├─ 과정 정보 입력
  ├─ NCS 능력단위 입력
  └─ 교재 파일 업로드
        ↓
[FastAPI Backend]
  ├─ 파일 파싱
  ├─ 문서 chunking
  ├─ embedding 생성
  ├─ vector DB 검색
  ├─ LLM 생성 요청
  ├─ 검수 상태 저장
  └─ DOCX/PPTX export
        ↓
[RAG / Agent Layer]
  ├─ 교안 생성 템플릿
  ├─ 실습 생성 템플릿
  ├─ 평가 문항 생성 템플릿
  └─ 근거 출처 매핑
        ↓
[산출물]
  ├─ 강의 교안 DOCX
  ├─ 실습 과제 DOCX
  ├─ 평가 문항 DOCX
  └─ 선택: PPTX 요약본
```

### 15.1 주요 컴포넌트

| 컴포넌트 | 역할 |
|---|---|
| Input Form | 과정명, 차시명, 학습목표, NCS 능력단위 입력 |
| Material Ingestor | PDF/DOCX/TXT 자료를 텍스트로 변환 |
| Retriever | 입력 주제와 관련 있는 문단 검색 |
| Generator | 교안·실습·평가 문항 생성 |
| Citation Mapper | 생성 항목별 근거 문단 연결 |
| Review UI | 강사가 결과를 수정하고 승인 |
| Exporter | DOCX/PPTX 파일 생성 |
| Audit Log | 입력, 검색 결과, 프롬프트, 생성 결과, 수정 이력 저장 |

### 15.2 데이터 흐름

1. 사용자가 교재와 커리큘럼 정보를 입력한다.
2. 시스템이 교재 텍스트를 chunk 단위로 분리한다.
3. 각 chunk에 embedding을 생성해 vector DB에 저장한다.
4. 차시명·학습목표·NCS 능력단위를 query로 사용해 관련 chunk를 검색한다.
5. 검색된 근거만 LLM 프롬프트에 넣는다.
6. LLM은 정해진 JSON 구조로 교안·실습·평가를 생성한다.
7. 시스템은 각 결과에 `근거 문단 ID`를 붙인다.
8. 강사는 결과를 수정하거나 승인한다.
9. 승인된 결과를 DOCX/PPTX로 export한다.

## 19. 1개월 개발 로드맵

### Week 1: 기획·데이터·RAG 최소 골격

목표: 입력 자료를 넣고 검색 가능한 상태까지 만든다.

| 작업 | 산출물 |
|---|---|
| 요구사항 확정 | MVP 범위표, 제외 기능표 |
| 샘플 데이터 준비 | 커리큘럼 1개, NCS 텍스트 1개, 교재 일부 |
| 파일 파싱 구현 | PDF/DOCX/TXT 중 최소 TXT + PDF |
| chunking 구현 | chunk ID, 원문 위치, 문서명 저장 |
| vector DB 연결 | Chroma 또는 대체 vector store |
| retrieval test | 차시명 입력 시 관련 문단 검색 결과 |

완료 기준:

- 파일 업로드 후 관련 문단 3~5개를 검색할 수 있다.
- 검색 결과에 문서명과 chunk ID가 포함된다.

### Week 2: 생성 Agent 구현

목표: 검색 근거를 사용해 1개 차시 패키지를 생성한다.

| 작업 | 산출물 |
|---|---|
| 프롬프트 템플릿 설계 | 교안, 실습, 평가별 템플릿 |
| 구조화 출력 설계 | JSON schema |
| LLM 호출 연결 | 선택한 모델 API 연동 |
| 교안 생성 | 도입·전개·정리 구조 |
| 실습 생성 | 시나리오·절차·제출물·루브릭 |
| 평가 생성 | 객관식 5문항 + 실습형 1개 |

완료 기준:

- 같은 입력에서 교안·실습·평가가 한 번에 생성된다.
- 생성 결과에 citation ID가 포함된다.

### Week 3: HITL 검수와 Export

목표: 강사가 결과를 수정하고 최종 문서로 저장한다.

| 작업 | 산출물 |
|---|---|
| 검수 UI 구현 | 결과 수정 화면 |
| 승인 상태 저장 | draft/reviewed/approved |
| DOCX 템플릿 구성 | 표지, 교안, 실습, 평가 섹션 |
| DOCX export | 다운로드 파일 |
| 생성 로그 저장 | 입력, 검색 문단, 프롬프트, 결과 |

완료 기준:

- 강사가 결과를 수정하고 승인할 수 있다.
- 승인된 결과가 DOCX로 다운로드된다.
- 생성 로그 파일이 남는다.

### Week 4: 품질 개선·발표 준비

목표: 시연 안정성과 기획서 완성도를 높인다.

| 작업 | 산출물 |
|---|---|
| PPTX export 선택 구현 | 시간이 남을 때만 |
| 근거 누락 검사 | citation 없는 항목 표시 |
| 결과 품질 평가 | 평가표와 예시 결과 |
| README 정리 | 실행 방법, 환경, 한계 |
| 발표 자료 구성 | 5분 데모 흐름 |

완료 기준:

- 1개 샘플 과정으로 end-to-end 시연이 가능하다.
- 문서 산출물과 로그가 저장된다.
- 한계와 추후 계획이 명시된다.

### 19.5 마일스톤 KPI

| KPI | 측정 방식 | MVP 목표 |
|---|---|---|
| E2E 성공률 | 샘플 1개 과정 기준 업로드→생성→검수→DOCX 다운로드 완료 여부 | 데모 시나리오 1개 성공 |
| 근거 연결률 | 생성 항목 중 citation ID가 붙은 항목 비율 | 핵심 항목 90% 이상 |
| 문서 생성 성공률 | 승인된 패키지에서 DOCX 파일 생성 여부 | 100% |
| 수정 가능성 | 생성 결과를 UI 또는 Markdown/JSON에서 수정 가능한지 | 강사 검수 후 저장 가능 |
| 발표 적합성 | 5분 발표에서 문제→해결→데모→로드맵 설명 가능 여부 | 7장 내외 발표 구성 가능 |

## 24. 한계와 추후 계획

### MVP 한계

- 전체 과정이 아니라 1개 차시 중심이다.
- 자동 채점과 학습자용 LMS 기능은 없다.
- NCS 데이터는 자동 연동이 아니라 수동 입력이다.
- 생성 품질은 교재 품질과 검색 결과에 크게 의존한다.
- PPTX export는 시간에 따라 선택 기능이 될 수 있다.

### 추후 확장

| 단계 | 확장 방향 |
|---|---|
| 2차 MVP | 다중 차시 일괄 생성, PPTX 안정화 |
| 3차 MVP | 문제은행, 난이도 분석, 문항 중복 검사 |
| 베타 | LMS 연동, SCORM/xAPI export 검토 |
| 상용화 | 기관 계정, 강사별 템플릿, 협업 검수, 보안 감사 |

## 25. 추천 결론

이 프로젝트는 처음부터 “강의 운영 전체를 자동화하는 AI Agent”로 접근하면 한 달 안에 완성하기 어렵다.  
따라서 MVP는 다음 한 문장으로 좁히는 것이 좋다.

> **교재와 커리큘럼을 근거로 1개 차시의 교안·실습·평가 초안을 만들고, 강사가 검수한 뒤 DOCX로 내려받는 RAG 기반 강의 운영 보조 Agent.**

이 범위는 원문 요구사항의 핵심인 RAG, HITL, 교안·실습·평가 생성, 문서 다운로드를 모두 포함하면서도 개인 프로젝트로 구현 가능한 수준이다. 상용 서비스들은 이미 유사한 기능을 제품화하고 있고, Dify·RAGFlow·Moodle·H5P·LangGraph·FastAPI 같은 오픈소스/프레임워크가 설계 근거를 뒷받침한다.

## 0. 현재 검증 상태

2026-07-21 기준 로컬 기준선과 실제 운영 연동 검증 체인이 구성되어 있으며, 데이터셋 43개 chunk, retrieval gold 10개, generation gold 3개 기준 검증을 수행한다. `--require-live-rag`는 Supabase pgvector의 실제 검색 결과를 평가하고, `--require-real-llm`은 LiteLLM 실제 응답과 구조화 출력 적용률을 검증한다.

동일 날짜 최종 실측에서는 Supabase와 실제 LiteLLM 호출을 모두 사용해 전체 판정 `PASS`를 기록했다. 검색은 Hit Rate/MRR/필수 개념 충족률 `1.0000`, Context Precision/Recall `0.8333`, nDCG@K `0.8642`였고, 생성은 3건 모두 구조화 출력·근거 연결·NCS 정합성·평가 품질 게이트를 통과했다. 상세 근거는 [`MVP 품질 평가 결과`](../04_validation/01_MVP_품질_평가_결과.md)에 보존한다.

현재 자동 검증의 핵심 명령은 다음과 같다.

```powershell
python -m unittest discover -s tests
python scripts\validate_mvp_dataset.py
python scripts\run_mvp_verification.py --output-dir outputs\eval --demo-case-id g003
python scripts\run_mvp_verification.py --require-live-rag --require-real-llm --output-dir outputs\eval-live
```

## 1. 목적

이 문서는 MVP 구현 결과가 실제로 동작하고, 생성 결과가 교재·NCS 근거에 기반하며, 자연어 수정 후에도 활용 가능한 품질인지 확인하기 위한 검증 절차를 정의한다.

검증은 다음 4가지를 구분한다.

1. 기능 검증
2. RAG 검색 검증
3. 생성 품질 검증
4. 실증 검증

---

## 2. 검증 원칙

| 원칙 | 설명 |
|---|---|
| 외부 실행 신호 기반 | “잘 될 것 같다”가 아니라 실행 로그와 파일 산출물로 판단 |
| 데이터셋 고정 | 동일한 샘플 데이터셋으로 반복 검증 |
| citation 우선 | 생성물의 근거 연결 여부를 핵심 지표로 둔다 |
| 사람 검수 포함 | 교육 품질은 자동 점수만으로 닫지 않는다 |
| 로그 보존 | 입력, 검색 chunk, 프롬프트, 생성 결과, 수정 이력을 남긴다 |

## 3. 검증 환경

| 항목 | 값 |
|---|---|
| OS | Windows 로컬 개발 환경 |
| Python | 3.11 이상 |
| API | FastAPI |
| UI | Lovable 웹 UI 또는 FastAPI Swagger UI |
| Vector DB | Supabase Postgres + pgvector, 로컬 테스트는 InMemory |
| Export | python-docx, python-pptx |
| 데이터셋 | `data/processed/` 43개 chunk와 `data/gold/` 평가 fixture 기준 MVP dataset |

실행 파라미터는 `config.yaml`에 명시한다. 코드 내부 암묵 기본값으로 검증하지 않는다.

## 4. 검증 단계

### 4.1 Level 0: 정적 검증

| 항목 | 방법 | 통과 기준 |
|---|---|---|
| 설정 파일 | `config.yaml` 확인 | 필수 파라미터 누락 없음 |
| schema | Pydantic model import | import error 없음 |
| 테스트 데이터 | data directory 확인 | raw, processed, gold 존재 |
| 라이선스 | `README_DATASET.md` 확인 | source URL과 license 기록 |

### 4.2 Level 1: 단위 검증

| 모듈 | 검증 항목 | 통과 기준 |
|---|---|---|
| parser_service | PDF/TXT 텍스트 추출 | 빈 문자열이 아니며 source metadata 유지 |
| chunk_service | chunk ID 생성 | 중복 ID 없음 |
| retrieval_service | top-k 검색 | 요청한 k개 이하 반환, metadata 포함 |
| generation_service | JSON 생성 | Pydantic schema 통과 |
| regeneration service | 자연어 수정 | 새 package ID, 원본 보존, 지시문 반영 |
| export_service | DOCX/PPTX 생성 | 파일 존재, 주요 섹션 포함 |
| audit_log_service | 로그 저장 | 입력·검색·프롬프트·결과 기록 |

### 4.3 Level 2: 통합 검증

시나리오:

```text
프로젝트 생성
→ 교재 업로드
→ chunking
→ vector DB upsert
→ retrieval
→ generation
→ regenerate(선택)
→ export.docx/export.pptx
→ log 확인
```

통과 기준:

- API 요청이 순서대로 성공한다.
- 최초 package status가 `generated`로 저장된다.
- 자연어 수정 후 새 package status가 `regenerated`가 된다.
- generated/regenerated 상태에서 DOCX/PPTX 파일이 생성된다.
- 로그 파일 또는 DB record가 생성된다.

### 4.4 Level 3: RAG 검색 검증

| 지표 | 측정 방법 | MVP 기준 |
|---|---|---|
| Top-k Hit Rate | gold expected chunk ID가 top-k 안에 있는 비율 | 0.70 이상 |
| MRR | 첫 관련 chunk 순위의 역수 평균 | 0.70 이상 |
| ID 기반 Context Precision | retrieved chunk ID와 expected chunk ID 비교 | 0.60 이상 |
| ID 기반 Context Recall | expected chunk ID 중 retrieved chunk에 포함된 비율 | 0.60 이상 |
| nDCG@K | 관련 chunk가 검색 상위에 배치되는 정도 | 관찰 지표, 기준선 기록 |
| 필수 개념 충족률 | gold의 `required_concepts`가 검색 문맥에 실제 포함되는 비율 | 0.70 이상 |
| 중복 chunk 비율 | top-k 내 동일 내용 반복 비율 | 20% 이하 |
| 빈 검색률 | 검색 결과가 0개인 query 비율 | 0% |

Ragas는 RAG 평가용 지표를 제공하며, context precision, context recall, faithfulness, response relevancy 등을 RAG 평가에 활용할 수 있다. 다만 MVP에서는 LLM 기반 평가 점수를 절대값으로 과신하지 않고, ID 기반 검증과 사람 검수를 병행한다.

### 4.5 Level 4: 생성 품질 검증

| 지표 | 측정 방법 | MVP 기준 |
|---|---|---|
| schema 통과율 | Pydantic validation | 100% |
| citation 연결률 | 핵심 생성 항목 중 citation ID 존재 비율 | 90% 이상 |
| citation-source 해소율 | 사용한 citation ID가 출처 상세 목록에서 모두 추적되는 비율 | 100% |
| 객관식 완성도 | 문항, 보기 4개, 정답, 해설 존재 | 5문항 모두 통과 |
| 실습형 평가 완성도 | 시나리오, 절차, 제출물, 루브릭 존재 | 모두 존재 |
| NCS 연결성 | 생성 항목이 NCS 능력단위와 연결되는지 | 주요 항목 80% 이상 |
| 출처 메타데이터 완성도 | 출처명, URL/파일/라이선스, 발췌문 보유 | 90% 이상 |
| 수업시간 일치도 | 강의 흐름 시간 합계와 표준 차시 시간 비교 | 90% 이상 |
| 문항 고유성 | 중복되지 않은 객관식 질문 비율 | 100% |
| 구조화 출력 적용률 | 실제 LLM JSON이 schema를 통과해 적용된 비율 | 실제 LLM 검증 시 100% |
| trace ID 보존율 | 생성 로그에서 Langfuse trace를 역추적할 수 있는 비율 | 실제 LLM 검증 시 100% |
| 문서 외형 품질 | 근거 단원 1회, UUID/내부 citation 미노출, 한글 레이블 확인 | DOCX/PPTX 모두 통과 |
| 근거 부족 표시 | 근거 없는 단정 대신 `근거 부족` 표시 | 위반 0건 |

### 4.6 Level 5: 실증 검증

| 항목 | 방법 | 통과 기준 |
|---|---|---|
| 사용자 이해 | 강사/예비 강사 2~3명 사용성 테스트 | 흐름을 설명 없이 수행 가능 |
| 신뢰도 | citation 표시 후 신뢰도 5점 척도 | 평균 4점 이상 목표 |
| 자연어 재생성 | 수정 지시 전후 package 비교 | 원본 보존, 새 package ID, 핵심 지시 반영 |
| 재사용 의향 | “다른 차시에도 써볼 의향” 질문 | 긍정 응답 확보 |
| 작성 시간 체감 | 기존 수작업 대비 비교 질문 | 시간 절감 체감 응답 확보 |

## 7. 사람 평가 루브릭

각 항목은 1~5점으로 평가한다.

| 평가 항목 | 1점 | 3점 | 5점 |
|---|---|---|---|
| 교안 적합성 | 차시 목표와 거의 무관 | 일부 관련 | 목표와 흐름이 명확히 연결 |
| 실습 적합성 | 수행 불가 | 일부 수정 필요 | 바로 실습 가능 |
| 평가 문항 품질 | 오류 많음 | 기본 구조는 있음 | 정답·해설·난이도 적절 |
| 근거 신뢰도 | 출처 불명확 | 일부 출처 있음 | 주요 내용에 근거 표시 |
| 수정 편의성 | 수정 어려움 | 수정 가능 | 강사가 쉽게 고칠 수 있음 |
| 문서 완성도 | 제출 어려움 | 보완 필요 | 운영 문서로 활용 가능 |

통과 기준:

- 전체 평균 3.5점 이상
- 근거 신뢰도 4점 이상
- 평가 문항 품질 3점 미만이면 재생성 또는 프롬프트 수정

## 9. 검증 실행 순서

1. 데이터셋 준비 여부 확인
2. 설정 파일 확인
3. parser/chunk 단위 검증
4. retrieval gold query 실행
5. mock LLM으로 schema·workflow 검증
6. 실제 LLM provider 연결
7. generation gold case 실행
8. citation coverage 확인
9. DOCX/PPTX export 확인
10. `scripts/run_mvp_verification.py`로 JSON/Markdown 통합 리포트 생성
11. 사용성 테스트 2~3명 진행
12. 결과 로그와 실패 분류 작성
13. README와 발표자료에 검증 결과 반영

로컬 결정론적 기준선 리포트 생성 명령:

```powershell
python scripts\run_mvp_verification.py --output-dir outputs\eval --demo-case-id g003
```

Supabase와 실제 LLM을 포함한 운영 연동 품질 리포트 생성 명령:

```powershell
python scripts\run_mvp_verification.py `
  --require-live-rag `
  --require-real-llm `
  --output-dir outputs\eval-live `
  --md-report docs\04_validation\01_MVP_품질_평가_결과.md
```

산출물:

- `outputs/eval/mvp_verification_report.json`
- `outputs/eval/mvp_verification_report.md`
- `outputs/eval/demo/*_lesson_package.docx`
- `outputs/eval/demo/*_lesson_package.pptx`

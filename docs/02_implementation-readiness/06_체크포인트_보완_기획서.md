# LessonPack AI 체크포인트 보완 기획서

작성일: 2026-07-21
대상 프로젝트: LessonPack AI

## 1. 제품 흐름 변경

기존의 별도 강사 검수·승인 단계는 제품 범위에서 제거했다. 현재 흐름은 다음과 같다.

```text
근거 기반 패키지 생성
→ 자연어 수정 요청(선택, 반복 가능)
→ DOCX/PPTX 다운로드
```

최초 생성은 `generated`, 자연어 수정으로 생성한 새 버전은 `regenerated` 상태를 사용한다. 두 상태 모두 별도 승인 없이 다운로드할 수 있다. 원본 패키지는 변경하지 않고 새 `package_id`를 발급한다.

이 결정으로 기존 체크포인트 3의 “별도 HITL 검수·승인 워크플로우”는 더 이상 제품 목표가 아니다. 대신 “사용자 자연어 지시 기반 수정, 원본 보존, 새 버전 추적”을 대체 검증 조건으로 사용한다.

## 2. 종합 판정

| 체크포인트 | 현재 판정 | 구현 근거 | 남은 과제 |
| --- | --- | --- | --- |
| 1. 커리큘럼 및 NCS 기반 표준 산출물 | 대부분 충족 | 학습목표·NCS 입력, 고정 패키지 스키마, 항목별 NCS alignment, 구조화 LLM JSON 검증 | 공식 NCS catalog code 대조 |
| 2. 교재 RAG, 할루시네이션 억제, 저작권·출처 | 대부분 충족 | Supabase pgvector 검색, citation ID 제한, source URL/license, 마지막 출처 단원 | 사용자 업로드 저작권 metadata 입력 |
| 3. 강사 검수·수정 HITL | 제품 범위에서 제외 | 별도 승인 상태와 검수 API 제거 | 외부 평가에서 필수 조건이면 범위 변경 승인 필요 |
| 대체 조건. 자연어 패키지 수정 | 구현 완료 | 기존 패키지+지시문+검색 근거를 LLM에 전달, 새 ID 발급, 원본 보존 | 실제 LLM 변경 대상 보존율 실증 |

## 3. 체크포인트 1: 표준 생성

### 현재 구현

- `ProjectCreate`가 과정명, 차시명, 학습자, 학습목표, NCS 능력단위를 입력받는다.
- `LessonPackage`는 교안, 실습, 객관식 5문항, 수행평가 1개를 고정 스키마로 관리한다.
- 교안 흐름, 실습, 객관식 문항, 수행평가에 `ncs_alignment`를 저장한다.
- provider JSON이 Pydantic 스키마와 허용 citation ID를 모두 통과할 때만 반영한다.
- DOCX/PPTX는 한국어 label과 표준 단원 순서를 사용한다.

### 보완 계획

| 우선순위 | 작업 | 완료 기준 |
| --- | --- | --- |
| P1 | NCS catalog resolver 추가 | 입력 unit code가 catalog에 존재하는지 검증 |
| P1 | NCS alignment coverage 지표 추가 | 핵심 산출물 90% 이상에 unit code 연결 |
| P2 | NCS 수행준거별 평가 문항 적합성 평가 | 사람 평가 평균 4/5 이상 |

## 4. 체크포인트 2: RAG와 출처

### 현재 구현

- 프로젝트 자료와 `mvp-dataset` baseline을 Supabase pgvector에서 검색한다.
- 검색 실행은 `retrieval_run_id`와 `trace_id`로 기록한다.
- 생성 프롬프트는 검색된 chunk만 전달한다.
- LLM 출력의 citation은 해당 검색 결과 ID의 부분집합이어야 한다.
- `evidence_sources`에 원천명, URL, license, 파일명, page, excerpt를 저장한다.
- DOCX/PPTX 본문에는 반복 citation을 넣지 않고 마지막 출처 단원에만 원천별로 정리한다.
- 다운로드명은 UUID 대신 `{수업명}_교안.docx`, `{수업명}_강의자료.pptx` 형식을 사용한다.

### 보완 계획

| 우선순위 | 작업 | 완료 기준 |
| --- | --- | --- |
| P1 | 업로드 저작권 metadata 입력 | 소유/이용 권한과 license 필드 필수화 |
| P1 | unsupported claim 평가 | 표본 산출물의 근거 없는 고유명사·수치 0건 |
| P2 | retrieval relevance threshold 조정 | context precision 목표 충족 |

## 5. 자연어 패키지 수정 대체 흐름

### API

```http
POST /api/packages/{package_id}/regenerate
```

```json
{
  "instruction": "실습 난이도를 낮추고 현장 예시를 추가해 주세요.",
  "top_k": 5,
  "include_baseline": true
}
```

### 처리 순서

1. 기존 패키지와 프로젝트를 조회한다.
2. 수정 지시문으로 관련 근거를 다시 검색한다.
3. 기존 패키지 전체, 지시문, 새 검색 근거를 LLM 프롬프트에 포함한다.
4. 지시받지 않은 단원은 유지하도록 요구한다.
5. 전체 패키지 JSON과 citation을 검증한다.
6. 새 `package_id`와 `regenerated` 상태로 저장한다.
7. 원본 패키지는 변경하지 않는다.

구조화 수정 응답이 검증에 실패하면 수정본을 저장하지 않고 `502`를 반환한다. 최초 생성용 deterministic fallback을 자연어 수정 성공으로 위장하지 않는다.

## 7. 검증 프로토콜

### 자동 검증

```powershell
python -m compileall -q src tests
python -m pytest -q
npm.cmd run build
```

필수 assertions:

- 최초 생성 상태는 `generated`다.
- 생성 직후 DOCX/PPTX export가 성공한다.
- 자연어 수정은 새 package ID를 반환한다.
- 수정 전 패키지는 내용과 상태가 바뀌지 않는다.
- 수정 지시와 기존 패키지가 LLM prompt에 포함된다.
- 수정 결과 상태는 `regenerated`다.
- 수정 결과도 즉시 export할 수 있다.
- generation log에 원본 package ID와 수정 지시가 남는다.
- DOCX/PPTX에 검수 이력 단원이 없다.
- citation은 검색된 chunk ID만 사용한다.

### 실제 LLM 실증

| 지표 | 측정 방법 | 목표 |
| --- | --- | --- |
| 수정 지시 반영률 | 지시 항목별 반영 여부 사람 평가 | 90% 이상 |
| 비대상 단원 보존율 | 수정 요청과 무관한 필드 비교 | 90% 이상 |
| 원본 보존 | 원본 package JSON hash 비교 | 100% |
| citation validity | 결과 citation이 retrieval set에 포함 | 100% |
| 구조 유효성 | Pydantic schema 통과 | 100% |
| export 성공률 | generated/regenerated DOCX/PPTX 생성 | 100% |

## 9. 운영 제약

- 현재 패키지 본문은 API 프로세스 메모리에 있으므로 재시작 후 조회되지 않는다.
- 운영 전 Supabase에 package 본문, source package ID, 생성 시각을 영속화해야 한다.
- 외부 제출 기준에서 별도 HITL 승인 상태가 필수라면 현재 제품 결정과 충돌한다. 그 경우 검수 기능을 숨은 자동 승인으로 복구하지 말고, 요구사항 변경 여부를 먼저 확정해야 한다.

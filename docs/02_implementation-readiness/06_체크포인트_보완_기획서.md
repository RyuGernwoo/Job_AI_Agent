## 3. 보완 목표

### 목표 A. NCS 기반 표준 생성의 증거 강화

현재 생성 결과는 구조는 표준화되어 있지만, 각 교안·실습·평가 항목이 어떤 NCS 수행준거에 대응하는지 명시적으로 남기지 않는다.

보완 방향:

1. NCS YAML을 로드하는 `NCSCatalog` 또는 `NCSResolver` 서비스를 추가한다.
2. `ProjectCreate.ncs_units` 입력값이 NCS catalog의 `unit_code`와 매칭되는지 검증한다.
3. `LessonPackage` 내부 항목에 `ncs_alignment` 필드를 추가한다.
4. 교안 도입·전개·정리, 실습 루브릭, 객관식 문항, 수행평가마다 연결된 NCS unit code와 performance criteria를 저장한다.
5. generation 평가 기준에 `ncs_alignment_required`를 추가한다.

완료 기준:

- 생성된 모든 lesson flow item, practice, assessment item에 최소 1개 이상의 NCS unit code가 연결된다.
- NCS unit code가 `data/raw/ncs/ncs_application_sw_programming.yaml`에 존재하지 않으면 생성 요청이 실패하거나 경고를 반환한다.
- generation 평가에서 NCS alignment coverage가 0.9 이상이다.

### 3.1 실제 테스트 산출물 품질 분석

분석 대상:

- `lessonpack-15df9b51-87d9-49f8-a8fe-11b107d0aba8.pptx`

분석 방법:

- DOCX paragraph/style 추출
- PPTX slide별 text block 추출
- citation, evidence source, license, NCS code, reviewer note, 한영 혼용 label 포함 여부 확인

확인된 구조:

| 항목 | DOCX | PPTX |
|---|---:|---:|
| 문서/슬라이드 규모 | 79개 non-empty paragraph | 8개 slide |
| citation 표시 | 있음 | 있음 |
| Evidence Sources 영역 | 있음 | 있음 |
| source URL 표시 | 없음 | 없음 |
| license 표시 | 없음 | 없음 |

주요 품질 문제:

| 문제 | 관찰 내용 | 영향 체크포인트 | 개선 방향 |
|---|---|---|---|
| citation 편중 | 모든 핵심 항목이 `doc001-p000-c075` 하나만 참조한다. | 1, 2 | top-k 근거 chunk를 항목별로 분배하고, citation diversity를 평가한다. |
| 출처 정보 부족 | Evidence Sources가 chunk ID만 보여주며 원천명, URL, license, excerpt가 없다. | 2 | `CitationDetail`을 패키지와 export에 포함한다. |
| NCS 근거 누락 | 산출물에 `2001020231`, `2001020230`, `2001020235` 같은 NCS unit code가 표시되지 않는다. | 1 | 산출물 항목별 `ncs_alignment` 필드를 추가한다. |
| LLM 응답 파싱 부재 | LLM 응답 전체를 구조화하지 않고, 전개 문단에 일부 요약처럼 잘라 붙인다. | 1, 2 | provider 응답을 JSON schema로 강제하거나, 실패 시 deterministic template fallback을 사용한다. |
| 전개 문단 품질 저하 | “인 테스트 코드를 작성한다”, CRUD 등 차시 목표와 약하게 연결된 문장이 들어갔다. | 1, 2 | retrieved chunk relevance threshold와 unsupported claim 검사를 추가한다. |
| 평가 문항 일반화 | 객관식 문항이 Python/NCS 역량 평가보다 MVP 사용법 확인에 가깝다. | 1 | 문항 생성 prompt와 template을 NCS 수행준거·학습목표 기반으로 재작성한다. |
| HITL 흔적 부재 | 승인 상태는 있으나 reviewer notes, 수정 이력, 수정 전후 내용이 export에 없다. | 3 | review event와 package edit history를 저장하고 export에 요약한다. |
| 한영 label 혼용 | `Learning Objectives`, `Submission`, `Answer`, `Explanation`, `Citations`가 그대로 노출된다. | 1, 3 | 한국어 export label dictionary를 적용한다. |
| PPTX 정보 과밀 | Practice slide에 긴 문단과 절차가 한 장에 몰려 가독성이 낮다. | 3 | PPTX는 교안/실습/평가/근거를 더 작은 slide 단위로 분할한다. |

품질 개선 우선순위:

1. `CitationDetail`과 NCS alignment를 먼저 패키지 모델에 포함한다.
2. export template을 한국어 label과 상세 근거 표시 중심으로 바꾼다.
3. LLM 출력은 자유 텍스트가 아니라 JSON schema 기반으로 받는다.
4. 평가 문항은 “서비스 사용법”이 아니라 “학습목표/NCS 수행준거”를 직접 측정하도록 생성 규칙을 바꾼다.
5. PPTX는 한 슬라이드당 핵심 메시지 3~5개 이하로 제한한다.

## 4.2 체크포인트 2 보완: RAG, 출처, 저작권 준수

### 4.2.1 citation을 ID에서 출처 정보로 확장

현재 export는 citation ID만 표시한다. 제출 체크포인트에서는 강사가 원천을 확인할 수 있어야 하므로 `CitationDetail` 스키마를 추가한다.

```python
class CitationDetail(BaseModel):
    chunk_id: str
    source_name: str
    source_url: str | None = None
    license: str | None = None
    source_file: str | None = None
    page: int | None = None
    excerpt: str
```

보완 방식:

1. `GenerateRequest` 또는 generation service에서 retrieved chunk metadata를 citation detail로 변환한다.
2. `LessonPackage`에 `evidence_sources: list[CitationDetail]` 필드를 추가한다.
3. DOCX/PPTX export의 Evidence Sources 영역에 chunk ID만이 아니라 원천명, URL, license, excerpt를 표시한다.
4. API 응답에서도 프론트엔드가 근거 패널을 만들 수 있도록 citation detail을 반환한다.

### 4.2.2 할루시네이션 억제 규칙 강화

현재 생성 프롬프트는 retrieved evidence를 포함하지만, 생성 결과가 evidence 밖으로 벗어났는지 강하게 검증하지 않는다.

보완 규칙:

- 생성 프롬프트에 “근거 chunk에 없는 고유명사, 수치, 법령, 인증명은 생성하지 말 것”을 명시한다.
- `generation_evaluation.py`에 unsupported claim 점검 항목을 추가한다.
- MVP 범위에서는 LLM-as-judge 대신 다음 단순 규칙부터 적용한다.
  - citation 없는 항목은 실패
  - citation ID가 retrieved chunk에 없으면 실패
  - 출처 URL/license가 없는 chunk를 사용하면 경고
  - 금지어 또는 무근거 표현을 발견하면 `needs_revision`

### 4.2.3 저작권 확인 흐름 추가

교재 업로드 시 사용자가 자료 사용 권한을 확인하는 필드가 없다. 보유 교재 RAG를 주장하려면 업로드 단계에서 최소한의 저작권 확인 메타데이터가 필요하다.

추가 입력:

| 필드 | 설명 |
|---|---|
| `license_type` | public, open_license, owned_by_instructor, institution_owned, unknown |
| `usage_attestation` | 강사가 교육 목적 활용 권한을 확인했는지 여부 |
| `source_url` | 공개 자료일 경우 원천 URL |
| `copyright_note` | 제한 사항 또는 내부 자료 표시 |

API 보완:

- `POST /api/projects/{project_id}/materials` 업로드 metadata 확장
- `MaterialChunk.metadata`에 license/copyright/source_url 저장
- unknown license는 generation 가능하더라도 export 시 “권한 확인 필요” 경고 표시

완료 기준:

- export 파일의 Evidence Sources에 모든 citation의 source name, URL 또는 source file, license note가 표시된다.
- 사용 권한 확인이 없는 업로드 자료는 `review_status=license_needs_review`로 표시된다.
- generation 평가 리포트에 citation coverage와 source metadata coverage가 함께 기록된다.

## 4.3 체크포인트 3 보완: HITL 검수·수정 워크플로우

### 4.3.1 현재 한계

현재 `ReviewPatch`는 다음 두 필드만 가진다.

- `status`
- `reviewer_notes`

그러나 `apply_review_patch()`는 reviewer notes를 `LessonPackage`에 저장하지 않고 상태만 바꾼다. 또한 교안, 실습, 평가 본문을 수정할 수 있는 patch 구조가 없다. 따라서 “검수”는 가능하지만 “검수·수정 워크플로우 완비”라고 보기는 어렵다.

### 4.3.2 수정 가능한 패키지 모델

추가 스키마:

```python
class ReviewEvent(BaseModel):
    event_id: str
    package_id: str
    from_status: PackageStatus
    to_status: PackageStatus
    reviewer_name: str | None = None
    reviewer_notes: str
    changed_fields: list[str] = []
    created_at: datetime

class PackageEditPatch(BaseModel):
    lesson_plan: LessonPlan | None = None
    practice: Practice | None = None
    assessment: Assessment | None = None
    edit_reason: str

class ReviewedLessonPackage(LessonPackage):
    reviewer_notes: str | None = None
    review_history: list[ReviewEvent] = []
```

### 4.3.3 API 보완

| API | 목적 |
|---|---|
| `PATCH /api/packages/{package_id}` | 교안·실습·평가 본문 수정 |
| `PATCH /api/packages/{package_id}/review` | reviewed, approved, needs_revision 상태 변경 |
| `GET /api/packages/{package_id}/review-history` | 검수 이력 조회 |
| `POST /api/packages/{package_id}/regenerate` | 반려 사유 기반 재생성 |

### 4.3.4 상태 전이 보완

권장 상태 흐름:

```text
draft
→ reviewed
→ approved
→ exported

draft
→ needs_revision
→ regenerated
→ reviewed
→ approved

reviewed
→ needs_revision
→ regenerated
```

상태 규칙:

- `approved` 이전 export 금지는 유지한다.
- `needs_revision`으로 변경할 때는 reviewer notes가 필수다.
- `regenerated` 패키지는 이전 package ID와 parent package ID를 기록한다.
- 모든 상태 전이는 `ReviewEvent`로 저장한다.

### 4.3.5 UI 보완

Lovable UI 또는 Streamlit UI에 다음 요소를 추가한다.

- 교안·실습·평가 탭별 inline editor
- citation detail 패널
- 항목별 “근거 확인 완료” 체크박스
- `needs_revision`, `reviewed`, `approved` 버튼
- reviewer notes 입력
- 수정 전/후 변경 요약 표시
- 승인 전 export 비활성화 유지

완료 기준:

- 강사가 생성된 교안, 실습, 평가 문항 본문을 수정할 수 있다.
- 수정 내용이 API 응답과 export 파일에 반영된다.
- reviewer notes와 review history가 패키지에 보존된다.
- 반려 후 재생성 또는 재검수 상태로 이동할 수 있다.

## 8. 최종 완료 정의

체크포인트 3종을 모두 충족했다고 판단하려면 다음 조건을 만족해야 한다.

- 커리큘럼, 학습목표, NCS 능력단위가 생성 프롬프트뿐 아니라 결과 항목별 metadata로 남는다.
- 교안, 실습, 평가 문항이 동일한 템플릿 버전과 NCS alignment 규칙으로 생성된다.
- LLM 응답은 JSON schema로 파싱되며, 파싱 실패 시 품질이 낮은 자유 텍스트를 그대로 export하지 않는다.
- 평가 문항은 서비스 사용법이 아니라 학습목표와 NCS 수행준거를 검증한다.
- RAG 검색 근거가 chunk ID, 원천명, URL 또는 파일명, license note, excerpt로 표시된다.
- 모든 핵심 항목이 하나의 citation에 과도하게 몰리지 않고, 최소 top-k 근거가 항목별로 분배된다.
- 저작권 확인이 필요한 자료는 명시적으로 경고된다.
- 강사가 산출물 본문을 수정할 수 있고, 수정 내용이 저장·조회·export된다.
- 검수 notes와 상태 전이 이력이 남는다.
- 승인 전 export는 계속 차단된다.
- DOCX/PPTX export는 한국어 label을 사용하고, PPTX는 슬라이드별 정보 과밀을 검사한다.
- 자동 테스트와 MVP 검증 리포트에서 위 항목이 수치로 확인된다.

## 9. 결론

현재 LessonPack AI는 한 달 MVP 기준으로 핵심 end-to-end 흐름을 갖춘 상태다. 다만 제출 체크포인트 기준에서는 다음 세 가지를 보완해야 한다.

1. NCS 수행준거를 산출물 항목별로 명시하는 구조
2. citation ID를 실제 출처·저작권 정보까지 확장하는 구조
3. 강사가 본문을 수정하고 이력을 남길 수 있는 HITL 구조

따라서 다음 개발 단계는 신규 기능 확장보다 “근거성, 출처성, 검수성”을 증명 가능한 데이터 구조와 테스트로 닫는 데 집중한다.

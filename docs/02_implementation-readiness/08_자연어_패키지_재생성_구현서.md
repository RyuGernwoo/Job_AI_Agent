# 자연어 패키지 재생성 구현서

## 1. 변경 목적

LessonPack AI의 강사 검수 및 승인 단계를 제거하고 다음 흐름으로 단순화한다.

```text
프로젝트 생성
→ 자료 업로드
→ 근거 검색
→ 패키지 생성
→ 자연어 수정 요청(선택, 반복 가능)
→ DOCX/PPTX 다운로드
```

생성된 패키지는 별도 승인 없이 즉시 다운로드할 수 있다. 자연어 수정은 기존 패키지를 직접 변경하지 않고, 기존 패키지와 수정 지시문을 LLM에 함께 전달해 새 `package_id`로 전체 패키지를 재생성한다.

## 2. 상태 모델

| 상태 | 의미 | 다운로드 |
| --- | --- | --- |
| `generated` | 최초 생성 완료 | 가능 |
| `regenerated` | 자연어 수정 요청으로 새 버전 생성 완료 | 가능 |
| `exported` | 내보내기 완료 상태를 별도로 기록할 때 사용 | 가능 |

`draft`, `reviewed`, `approved`, `needs_revision` 상태와 승인 전 export 차단 규칙은 사용하지 않는다.

## 3. 자연어 재생성 API

### 요청

```http
POST /api/packages/{package_id}/regenerate
Content-Type: application/json
```

```json
{
  "instruction": "실습 난이도를 초급으로 낮추고 도입 문장을 쉽게 바꿔 주세요.",
  "top_k": 5,
  "include_baseline": true
}
```

### 응답

```json
{
  "package": {
    "package_id": "new-package-id",
    "project_id": "project-id",
    "status": "regenerated"
  },
  "source_package_id": "previous-package-id",
  "retrieval_run_id": "retrieval-run-id",
  "trace_id": "trace-id"
}
```

### 처리 규칙

1. `{package_id}`로 기존 패키지와 프로젝트를 조회한다.
2. 수정 지시문을 RAG 검색 질의에 포함해 관련 근거를 다시 검색한다.
3. 기존 패키지 JSON, 자연어 지시문, 새 검색 근거를 하나의 생성 프롬프트에 포함한다.
4. LLM에는 지시받지 않은 단원은 유지하고 지시 대상만 변경하도록 요구한다.
5. LLM은 교안·실습·평가 전체 JSON을 반환한다.
6. citation ID와 구조화 출력 스키마를 검증한다.
7. 새 UUID와 `regenerated` 상태로 저장하고 원본 패키지는 유지한다.

재생성 응답이 구조화 스키마 또는 citation 제약을 지키지 않으면 fallback 결과를 새 수정본으로 위장하지 않고 `502`를 반환한다.

## 4. 프롬프트 안전 규칙

- 자연어 입력은 콘텐츠 수정 지시로만 취급한다.
- 출력 JSON 스키마와 citation 제약을 지시문이 변경할 수 없다.
- 새 검색 결과에 포함된 chunk ID만 citation으로 사용할 수 있다.
- 지시받지 않은 단원과 세부 내용은 가능한 한 유지한다.
- 기존 패키지를 직접 덮어쓰지 않는다.
- 생성 로그에 `source_package_id`와 `revision_instruction`을 기록한다.

## 5. 프론트엔드 계약

- 전체 단계는 프로젝트, 교재 업로드, 근거 검색, 패키지 생성, 다운로드의 5단계다.
- 패키지 생성 성공 직후 다운로드 단계로 이동할 수 있다.
- 생성 화면 안에서 자연어 수정 요청을 반복할 수 있다.
- 수정 요청은 현재 `package_id`를 `/regenerate` 엔드포인트에 전달한다.
- 응답의 새 패키지로 현재 화면을 교체하고 이전 생성 로그 캐시는 비운다.
- `autoApprove`와 검수 API 호출은 사용하지 않는다.
- 다운로드 버튼은 `generated`, `regenerated`, `exported` 패키지에서 활성화한다.
- 다운로드명은 API `Content-Disposition` 헤더를 우선 사용한다.

## 6. 오류 처리

| HTTP 상태 | 조건 | 사용자 안내 |
| --- | --- | --- |
| 404 | 원본 패키지를 찾을 수 없음 | 패키지를 다시 생성하도록 안내 |
| 422 | 수정 지시문이 비었거나 검색 근거가 없음 | 지시 구체화 또는 자료 추가 안내 |
| 502 | LLM 수정 응답이 구조화 검증 실패 | 다시 시도하도록 안내하고 원본 유지 |
| 503 | Supabase 등 영속 저장소 오류 | 잠시 후 재시도 안내 |

## 7. 검증 항목

- 최초 생성 상태가 `generated`인지 확인한다.
- 최초 생성 직후 DOCX/PPTX export가 200인지 확인한다.
- 재생성 요청이 새 `package_id`를 반환하는지 확인한다.
- 원본 패키지 내용과 상태가 바뀌지 않는지 확인한다.
- 자연어 지시문과 기존 패키지 내용이 LLM 프롬프트에 포함되는지 확인한다.
- 재생성 상태가 `regenerated`인지 확인한다.
- 재생성 결과도 즉시 export 가능한지 확인한다.
- generation log가 원본 패키지 ID와 지시문을 기록하는지 확인한다.
- DOCX/PPTX에 검수 이력 단원이 포함되지 않는지 확인한다.

## 9. 현재 제약과 후속 작업

- API 프로세스가 재시작되면 메모리에 보관된 패키지를 다시 조회할 수 없다. 운영 단계에서는 패키지 본문과 버전 관계를 Supabase에 영속화해야 한다.
- `source_package_id`를 부모로 갖는 버전 계보 테이블을 추가하면 이전 버전 조회와 되돌리기를 구현할 수 있다.
- 자연어 부분 수정 품질은 실제 LLM 실증에서 변경 대상 보존율과 비대상 단원 보존율로 추가 평가해야 한다.
